# Рефакторинг usecase-модуля question/commands под новый домен

## Контекст

Необходимо отрефакторить application usecase-модуль создания и изменения вопросов.

Модуль находится в пакете:

```text
internal/application/usecases/question/commands
```

Домен вопроса был изменён, поэтому текущий application-код использует устаревшие типы, методы и builder-функции.

Нужно проанализировать актуальную кодовую базу и привести usecase-команды в соответствие с новым доменом.

---

# Цель 1. Адаптировать CreateUseCase под новый домен

Необходимо отрефакторить:

```go
CreateUseCase
CreateInput
Execute
createQuestion
build-функции, если они используются созданием вопроса
```

Текущая структура `CreateInput` может содержать устаревшие поля:

```go
Attachment        *AttachmentInput
SelectableOptions []SelectableOptionInput
SequenceOptions   []SequenceOptionInput
MatchingPairs     []MatchingPairInput
TypedBlanks       []TypedBlankInput
ShortVariants     []ShortVariantInput
```

Необходимо:

* проанализировать новый домен вопроса;
* определить, какие input-типы и поля больше не нужны;
* удалить устаревшие input-типы;
* удалить устаревшие поля из `CreateInput`;
* обновить создание вопросов под новые конструкторы домена;
* сохранить бизнес-смысл сценария создания вопроса;
* не оставлять мёртвый код.

---

# Цель 2. Удалить устаревшие input-типы

Необходимо проанализировать и удалить input-типы, которые больше не используются из-за изменения домена.

Текущие кандидаты на удаление:

```go
type AttachmentInput struct {
	MediaType string
	FileName  string
	SizeBytes int64
}

type SelectableOptionInput struct {
	Text      string
	IsCorrect bool
}

type SequenceOptionInput struct {
	Text string
}

type MatchingPairInput struct {
	Prompt string
	Match  string
}

type TypedBlankInput struct {
	Placeholder string
	Variants    []string
}

type ShortVariantInput struct {
	Text string
}
```

Требования:

* удалить только реально неиспользуемые типы;
* если часть типов всё ещё нужна новому домену, адаптировать их;
* не удалять типы механически без анализа использования;
* после удаления обновить все связанные build-функции, тесты и usecase-команды.

---

# Цель 3. Удалить устаревшие usecase-методы изменения доменных частей

Необходимо проанализировать usecase-команды изменения вопроса и удалить/адаптировать методы, которые больше не нужны из-за нового домена.

Особое внимание уделить:

```go
ChangeAttachmentUseCase
RemoveAttachmentUseCase
ChangeSelectableOptionsUseCase
ChangeSequenceOptionsUseCase
ChangeMatchingPairsUseCase
ChangeTypedContentUseCase
ChangeShortVariantsUseCase
```

Требования:

* если новый домен больше не поддерживает отдельное изменение части вопроса — удалить соответствующий usecase;
* если новый домен поддерживает сценарий иначе — адаптировать usecase под новый API домена;
* удалить устаревшие input-структуры;
* удалить устаревшие builder-функции;
* удалить или обновить тесты;
* не оставлять неиспользуемый код.

# Цель 5. Обновить builder-функции

Необходимо найти и проанализировать все builder-функции внутри пакета `commands`, например:

```go
buildAttachment
buildSelectableOptions
buildSequenceOptions
buildMatchingPairs
buildTypedBlanks
buildShortVariants

```

Требования:

* удалить builder-функции, которые больше не нужны;
* адаптировать оставшиеся builder-функции под новый домен;
* не дублировать доменную валидацию в application-слое;
* application-слой должен только преобразовывать input в доменные объекты;
* инварианты должны оставаться в домене.

# Цель 6. Русифицировать ошибки

Необходимо перевести ошибки пакета `commands` на русский язык.

Требования:

* ошибки должны быть понятными и содержать контекст;
* использовать wrapping через `%w`;
* не терять исходную ошибку;
* не менять sentinel errors;
* не использовать английские сообщения вида `save question`, `create question title`, `question is not matching`.

Примеры стиля:

```go
fmt.Errorf("создание заголовка вопроса: %w", err)
fmt.Errorf("сохранение вопроса: %w", err)
fmt.Errorf("%w: вопрос не является вопросом на сопоставление", common.ErrInvalidInput)
```

# Цель 7. Обновить тесты

Необходимо добавить или обновить тесты usecase-команд.

Тесты должны покрывать:

* успешное создание вопроса;
* ошибки проверки роли;
* ошибки парсинга типа вопроса;
* ошибки создания заголовка;
* ошибки создания доменного агрегата;
* ошибки загрузки вопроса;
* ошибки сохранения вопроса;
* сценарии устаревших методов, если они были адаптированы;
* отсутствие вызова repository при ранней ошибке;
* корректный ID в `Output`.

Требования:

* использовать table-driven tests;
* использовать fake/mock repository;
* проверять `errors.Is`, если используются sentinel errors;
* не тестировать внутреннюю реализацию домена повторно;
* тестировать orchestration application-слоя;
* не создавать хрупкие тесты, завязанные на несущественный текст ошибок.

# Цель 9. Проверить архитектурные проблемы

После рефакторинга необходимо отдельно проанализировать архитектуру пакета `commands`.

Проверить:

* не протекает ли infrastructure в application/domain;
* не дублируется ли доменная валидация;
* нет ли лишних интерфейсов;
* нет ли устаревших DTO;
* нет ли мёртвого кода;
* не нарушены ли границы usecase/application/domain;
* не появился ли god-usecase;
* не нужно ли разделить слишком крупные файлы.

Если будут найдены архитектурные проблемы — исправить только те, которые относятся к текущей задаче. Остальные описать в финальном отчёте без изменения unrelated-кода.

* Не менять домен без необходимости.

* Не выполнять unrelated-рефакторинг.
* Не добавлять новые абстракции без явной необходимости.
* Не добавлять лишние интерфейсы.
* Не использовать panic, кроме существующих constructor guard-проверок, если такой стиль уже принят в проекте.
* Не дублировать доменную валидацию в application-слое.
* Сохранять существующий стиль проекта.
