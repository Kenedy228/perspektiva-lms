# Рефакторинг пакета matching

## Контекст

Необходимо провести рефакторинг подпакета `matching/answer`.

Текущая реализация использует типы `PromptID` и `MatchID`.
Необходимо отказаться от этих типов и использовать `uuid.UUID`.

Рефакторинг должен быть локальным, без изменения бизнес-логики пакета, за исключением описанных изменений.

---

# Цель 1. Удалить типы PromptID и MatchID

Необходимо полностью отказаться от использования типов:
- `PromptID`
- `MatchID`

Вместо них должен использоваться `uuid.UUID`.

---

## Задача 1. Изменить структуру Pair

Структура `Pair` должна принять следующий вид:

```go
type Pair struct {
	PromptID uuid.UUID
	MatchID  uuid.UUID
}
```

## Задача 2. Изменить проверку ID в validatePairIDsNotEmpty

В функции:

```go
validatePairIDsNotEmpty
```

необходимо изменить логику проверки `PromptID` и `MatchID`.

Новая логика:

- если `PromptID == uuid.Nil`, возвращаем ошибку;
- если `MatchID == uuid.Nil`, возвращаем ошибку.

---

## Задача 3. Изменить сигнатуру метода Answer.AsMap

Необходимо изменить сигнатуру возвращаемого значения.

Было:

```go
map[PromptID]MatchID
```

Стало:

```go
map[uuid.UUID]uuid.UUID
```

---

# Цель 2. Улучшить диагностику ошибок

После выполнения рефакторинга необходимо улучшить ошибки пакета.

Требования:

- русифицировать текст ошибок;
- добавить более подробный контекст ошибок;
- использовать wrapping через %w;
- ошибки должны помогать понять причину возникновения проблемы.

Пример ожидаемого стиля:

```go
fmt.Errorf(
	"%w: количество пар не должно превышать %d штук (текущее количество - %d)",
	ErrInvalid,
	matching.MaxPairs,
	len(pairs),
)
```

---

# Цель 3. Улучшить тестовое покрытие

Необходимо добавить тесты:

- на экспортируемые методы;
- на неэкспортируемые функции;
- на сложные инварианты;
- на edge-case сценарии;
- на ошибки;
- на граничные значения.

Требования к тестам:

- использовать table-driven tests;
- избегать дублирования;
- тесты должны быть читаемыми;
- тестировать не только happy-path.

Особое внимание уделить:

- пустым UUID (uuid.Nil);
- дубликатам;
- пустым слайсам;
- превышению лимитов;
- некорректным pair combinations.

---

# Цель 4. Добавить документацию

Необходимо добавить godoc-комментарии на русском языке для:

- публичных структур;
- публичных методов;
- публичных функций;
- экспортируемых ошибок.

Требования:

- комментарии должны начинаться с имени сущности;
- комментарии должны описывать назначение и поведение;
- избегать формальных и бесполезных описаний.

---

Ограничения
- Не изменять бизнес-логику пакета, кроме явно описанных изменений.
- Не добавлять новые абстракции без необходимости.
- Не выполнять рефакторинг unrelated-кода.
- Не изменять API других пакетов без необходимости.
- Сохранить существующий стиль проекта.
- Не использовать panic.
- Не добавлять лишние интерфейсы.

---
