# Рефакторинг доменного пакета grading под новый question domain

## Контекст

Необходимо отрефакторить доменный пакет:

```text
internal/domain/grading
```

Пакет `question` и его подпакеты были изменены, поэтому grading-чекеры могут использовать устаревшие методы, типы, алиасы импортов и старые ошибки.

Нужно проанализировать актуальный домен вопросов и привести grading-код в соответствие с новым API.

---

# Цель 1. Проверить совместимость grading с новым question domain

Необходимо проанализировать чекеры:

```text
internal/domain/grading/matching
internal/domain/grading/selectable
internal/domain/grading/sequence
internal/domain/grading/short
```

Проверить:

* актуальны ли используемые методы доменных вопросов;
* актуальны ли используемые методы доменных ответов;
* не используются ли старые методы после рефакторинга домена;
* не используются ли устаревшие value objects;
* корректно ли работают проверки типов вопроса и ответа;
* корректно ли считается score.

Если API домена изменился — адаптировать grading под новый API.

---

# Цель 2. Убрать лишние алиасы импортов

Необходимо проверить импорты в grading-пакетах.

Пример текущего нежелательного стиля:

```go
question2 "gitflic.ru/lms/backend/internal/domain/question"
qmatching "gitflic.ru/lms/backend/internal/domain/question/matching"
qselectable "gitflic.ru/lms/backend/internal/domain/question/selectable"
qsequence "gitflic.ru/lms/backend/internal/domain/question/sequence"
```

Требования:

* убрать лишние алиасы импортов;
* оставить алиасы только там, где они реально нужны для устранения конфликта имён;
* привести импорты к читаемому и идиоматичному виду;
* сохранить понятность кода.

---

# Цель 3. Русифицировать ошибки

Необходимо проверить ошибки во всех grading-подпакетах и перевести их на русский язык.

Особое внимание уделить ошибкам:

```go
ErrInvalidQuestionType
ErrInvalidAnswerType
```

Требования:

* ошибки должны быть на русском языке;
* ошибки должны быть понятными;
* ошибки должны содержать контекст, если это уместно;
* не терять sentinel errors;
* если ошибка оборачивается — использовать `%w`;
* не оставлять английские сообщения.

Примеры ожидаемого стиля:

```go
fmt.Errorf("%w: ожидался вопрос на сопоставление", ErrInvalidQuestionType)
fmt.Errorf("%w: ожидался ответ на сопоставление", ErrInvalidAnswerType)
```

Если в текущей архитектуре используются только sentinel errors без контекста — можно русифицировать сами sentinel errors.

---

# Цель 4. Обновить и расширить тесты

Необходимо проверить существующие тесты grading-пакетов и обновить их под новый домен.

Тесты должны покрывать:

* успешную проверку ответа;
* неверный тип вопроса;
* неверный тип ответа;
* полностью правильный ответ;
* полностью неправильный ответ;
* частично правильный ответ;
* лишние ответы;
* недостаточное количество ответов;
* пустые ответы;
* порядок вариантов для sequence;
* неправильные пары для matching;
* лишние выбранные варианты для selectable;
* нормализацию короткого ответа для short.

Требования:

* использовать table-driven tests;
* проверять результат score;
* проверять ошибки через `errors.Is`, если используются sentinel errors;
* не тестировать повторно внутреннюю валидацию question domain;
* тестировать только поведение grading;
* избегать дублирования setup-кода.

---

# Цель 5. Добавить godoc на русском языке

Необходимо добавить или обновить godoc-комментарии на русском языке для публичного API grading-пакетов.

Документация нужна для:

* публичных структур `Checker`;
* публичных конструкторов `New`;
* публичных методов `Check`;
* публичных методов `Supports`;
* публичных опций, если они есть;
* экспортируемых ошибок.

Требования:

* комментарий должен начинаться с имени документируемой сущности;
* комментарий должен описывать назначение и поведение;
* избегать формальных комментариев без смысла.

---

# Цель 6. Проверить архитектурные нюансы

После рефакторинга необходимо проверить архитектуру grading-пакета.

Проверить:

* не дублирует ли grading доменную валидацию вопросов;
* не нарушает ли grading границы домена;
* нет ли зависимости от application или infrastructure;
* нет ли лишних интерфейсов;
* нет ли мёртвого кода;
* нет ли ненужных алиасов;
* не стоит ли вынести повторяющуюся логику создания score;
* не игнорируются ли ошибки `score.New` без причины.

Если архитектурные проблемы относятся к текущей задаче — исправить.
Если проблема крупнее текущего рефакторинга — описать её в финальном отчёте без unrelated-изменений.

---

# Ограничения

* Не менять application-слой.
* Не менять infrastructure-слой.
* Не менять transport-слой.
* Не менять question domain без необходимости.
* Не выполнять unrelated-рефакторинг.
* Не добавлять новые абстракции без явной необходимости.
* Не добавлять лишние интерфейсы.
* Не дублировать domain validation.
* Сохранять существующий стиль проекта.

* устаревшие обращения к question/answer API обновлены;
* лишние алиасы импортов удалены;
* ошибки переведены на русский язык;
* тесты обновлены и расширены;
* godoc добавлен;
* архитектурные проблемы проверены;
* проект успешно проходит `go test ./...` и `go vet ./...`.
