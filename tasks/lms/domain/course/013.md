# Задача для Codex: завершение рефакторинга course-модуля

## Цель

Довести рефакторинг `course`-модуля проекта `perspektiva-lms` до консистентного состояния.

Текущий рефакторинг выполнен частично:

* `Course` уже переведён на `blockIDs`;
* `Element` уже получил `CompletionMode`;
* `version` удалён из domain tree;
* добавлены новые content type'ы;
* добавлены docs и diagrams.

Но кодовая база ещё находится в промежуточном состоянии:

* часть imports/пакетов не совпадает;
* progress всё ещё зависит от `versionID`;
* repository/usecases всё ещё partially version-centric;
* остались compatibility-методы;
* project, вероятно, не собирается.

Нужно завершить рефакторинг без rewrite и без overengineering.

---

# Главные проблемы

## P1. Несогласованность content-пакетов

Сейчас:

* `helpers.go` импортирует:

    * `content/downloadfile`
    * `content/lecturematerial`
    * `content/test`
* но в repository/tree всё ещё присутствуют:

    * `attachment`
    * `quiz`
    * `slides`
    * `text`
    * `video`

Нужно привести content-модель к единому состоянию.

---

## P2. Progress всё ещё зависит от versionID

В `progress` всё ещё есть:

* `versionID`;
* `VersionID()`;
* `New(enrollmentID, versionID)`;
* `Restore(..., versionID, ...)`.

Это противоречит новой модели:

```txt
Course → Block → Element
```

без `version`.

---

## P3. Course содержит compatibility methods

В `Course` всё ещё есть:

* `VersionIDs()`;
* `AddVersionID()`;
* `RemoveVersionID()`;
* `HasVersion()`.

Это временные compatibility-хвосты.

---

## P4. Repository всё ещё version-centric

В PostgreSQL mapping и policy всё ещё используются:

* `course_versions`;
* `course_version_links`;
* `version_id`;
* `CanEnrollVersion`;
* joins через version-flow.

Нужно перевести active flow на:

* `course → blocks`;
* `block → elements`.

---

## P5. Use cases обновлены частично

DTO уже содержит:

* `AddBlockToCourse`;
* `MoveCourseBlock`;
* `AddElementToBlock`;
* `MoveBlockElement`.

Но часть use case'ов отсутствует или не завершена.

---

# Что нужно сделать

==================================================
ЧАСТЬ 1 — Завершить content model
=================================

## Целевая модель

Оставить только 3 бизнес-типа:

```txt
1. Test
2. LectureMaterial
3. DownloadFile
```

## Что сделать

Привести content packages к консистентному состоянию.

### Разрешённые content packages

```txt
content/test
content/lecturematerial
content/downloadfile
```

### Старые content packages

Найти:

* `attachment`
* `quiz`
* `slides`
* `text`
* `video`

И:

* либо удалить;
* либо пометить deprecated;
* либо оставить compatibility adapters временно.

Но active flow должен использовать только новую модель.

## LectureMaterial

Поддерживаемые виды:

```txt
- video
- pdf
```

## DownloadFile

Поддерживаемые файлы:

```txt
- image
- txt
- doc/docx
- ppt/pptx
- xls/xlsx
- pdf
- video
```

## Важно

Домен:

* НЕ должен зависеть от S3;
* НЕ должен зависеть от HTTP upload;
* НЕ должен зависеть от PostgreSQL.

Использовать существующий shared file VO.

==================================================
ЧАСТЬ 2 — Удалить versionID из Progress
=======================================

## Что менять

```txt
backend/internal/domain/course/progress
```

## Что сделать

Удалить:

* `versionID`;
* `VersionID()`;
* все constructor/repository зависимости от version.

## Целевая модель

```txt
Progress
  - enrollmentID
  - completedElementIDs / markers
```

или:

```txt
Progress
  - enrollmentID
  - courseID
  - completed markers
```

если courseID нужен явно.

## Реализовать

```go
MarkCompleted(elementID)
UnmarkCompleted(elementID)
IsCompleted(elementID)
CompletedCount()
Percent()
```

## Важно

Progress:

* НЕ должен знать структуру Course;
* НЕ должен знать Block;
* НЕ должен знать repository.

# Ограничения

НЕ делать:

* rewrite всего проекта;
* enterprise Java architecture;
* interfaces ради interfaces;
* reflection;
* service locator;
* generic abstraction hell.

НУЖНО:

* explicit Go code;
* минимальные abstraction;
* простые зависимости;
* deterministic behavior;
* сохранить DDD boundaries.

---

# Критерии готовности

Задача считается выполненной, если:

1. Active code больше не зависит от `course/version`.
2. Progress больше не зависит от `versionID`.
3. Content model консистентна.
4. Repository больше не version-centric.
5. HTTP routes больше не используют version.
6. Compatibility methods удалены.
7. Все tests проходят.
8. `go test ./...` проходит.
9. Диаграммы обновлены.
10. Код стал проще, а не сложнее.
